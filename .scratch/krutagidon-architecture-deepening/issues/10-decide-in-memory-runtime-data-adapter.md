# Решить, нужен ли полноценный in-memory Runtime Data Adapter

## What to build

Принять архитектурное решение после того, как `initializeGame` научится принимать готовый `LoadedDataPack`.

Нужно решить, достаточно ли этого Interface для tests, или проекту нужен полноценный in-memory Runtime Data Adapter с helper API для сборки маленьких executable data packs. Решение должно учитывать правило: один Adapter = гипотетический Seam, два Adapter = реальный Seam.

## Decision

Production in-memory Runtime Data Adapter сейчас не делать.

Issue 09 уже добавил самый маленький полезный production seam: `initializeGame` принимает готовый `LoadedDataPack`, а CLI и simulation runners продолжают использовать filesystem loader. Этого достаточно, чтобы отделить engine setup от чтения filesystem без введения второго поддерживаемого runtime data source.

Текущие основания относятся к удобству тестов, а не к production runtime need:

- `tests/setup.test.ts` проверяет, что setup из готового `LoadedDataPack` совпадает с filesystem setup.
- `tests/effective-values.test.ts` содержит один fixture-heavy test, который прямо выиграл от подготовки `LoadedDataPack` до `initializeGame`.
- Остальные мутации definitions сосредоточены в `tests/action-loop.test.ts`, плюс один helper в `tests/effective-values.test.ts`.
- Эти мутации в основном добавляют fixture definitions для card/token instances, которые тест вручную вставляет в уже созданный `GameState`: hand, market, deck, permanent zone или wizard property slot.

Подготовка `LoadedDataPack` до `initializeGame` помогает, когда сами setup-правила должны увидеть изменённые card/token definitions. Она не убирает потребность action-loop тестов лепить post-setup state, потому что эти тесты проверяют конкретные action states, а не загрузку initial setup data.

Shared tests helper в этом issue не добавляется. Повторяющийся local fixture setup есть, но повторяется паттерн "точечно пропатчить post-setup `GameState` для action test", а не "собрать reusable executable in-memory data pack". Generic helper сейчас либо будет слишком тонкой обёрткой над копированием map в `LoadedDataPack`, либо преждевременно закрепит post-setup state mutation как широкую абстракцию.

Follow-up AFK issue для production adapter не создаётся. В проекте всё ещё есть только один production adapter: filesystem loader. По правилу проекта один adapter = гипотетический seam, два adapters = реальный seam. Production in-memory adapter должен ждать реального production consumer, например UI/editor/sandbox/strategy runner, который собирает runtime data в памяти вне тестов.

Решение стоит пересмотреть, когда появится второй production consumer, которому нужно создавать или предоставлять runtime data без чтения filesystem.
